Ниже приведён структурированный и формализованный план начала разработки FinanceAI App. Он ориентирован на достижение работающего MVP-ядра в краткие сроки (2–3 недели) с минимальными рисками распыления усилий.

---

# 1. Цель начального этапа

Основная цель — не разработка полного продукта, а создание **работающего сквозного MVP-скелета**:

* мобильное приложение (базовый интерфейс)
* backend API
* база данных
* деплой на сервер
* взаимодействие mobile ↔ backend

Результат: пользователь может открыть приложение, выполнить действие (например, добавить транзакцию), и данные сохраняются и отображаются.

---

# 2. Общая архитектура разработки

Работа делится на три параллельных направления:

## 2.1 Backend (приоритет)

* бизнес-логика
* API
* база данных

## 2.2 Mobile (интерфейс)

* UI
* взаимодействие с API

## 2.3 Infrastructure (инфраструктура)

* сервер
* Docker
* CI/CD

---

# 3. Пошаговый план (2–3 недели)

## Этап 1. Backend-основа (Дни 1–4)

Цель: локально работающий API

### Задачи:

* создать backend (FastAPI)
* подключить PostgreSQL
* настроить Dockerfile и docker-compose
* создать файл `.env`
* реализовать базовый endpoint:

```
GET /health
```

* реализовать CRUD для транзакций:

```
POST /transactions
GET /transactions
```

---

## Этап 2. Локальное окружение (Дни 3–6)

Цель: стабильная разработка без ручных операций

### Задачи:

* запуск backend через Docker:

```
docker compose up
```

* включить автообновление (reload)
* подключить базу данных
* настроить миграции

---

## Этап 3. Первый деплой (Дни 5–7)

Цель: backend доступен извне

### Задачи:

* развернуть backend на VPS
* настроить docker compose на сервере
* (опционально) настроить nginx
* проверить доступность:

```
GET /health
```

---

## Этап 4. Базовый CI/CD (Дни 7–10)

Цель: автоматический деплой

### Задачи:

* настроить GitHub Actions:

  * запуск тестов
  * деплой через SSH

Результат:

```
git push → обновление на сервере
```

---

## Этап 5. Mobile skeleton (Дни 8–12)

Цель: базовое мобильное приложение

### Задачи:

* создать Flutter проект
* реализовать 2 экрана:

  * список транзакций
  * добавление транзакции
* подключить приложение к backend API

Результат: приложение можно запустить на устройстве и взаимодействовать с API

---

## Этап 6. Авторизация (Дни 10–14)

Цель: пользовательская система

### Задачи:

* регистрация и логин
* JWT-аутентификация
* хранение токена в приложении

---

## Этап 7. Минимальная AI-функция (Дни 12–16)

Цель: реализация ключевого дифференциатора

### Задачи:

* endpoint:

```
POST /ai/advice
```

* интеграция с AI API (OpenAI или аналог)
* передача контекста (последние транзакции)

---

# 4. Результат после 2–3 недель

Должен быть получен следующий функциональный минимум:

* backend развернут на сервере
* API доступен извне
* мобильное приложение запускается на устройстве
* пользователь может:

  * добавить транзакцию
  * увидеть список транзакций
* AI-эндпоинт отвечает
* настроен базовый деплой pipeline

---

Ваш текущий документ:

Однако для старта необходим **операционный план**, который:

* разбивает работу на короткие шаги
* даёт ежедневные задачи
* позволяет быстро получить работающий результат

Настоящий документ выполняет именно эту функцию.

---

# 6. Рекомендации по работе

При выполнении задач следует придерживаться следующих принципов:

1. Каждая задача должна приводить к проверяемому результату
2. Каждая новая функция должна быть тестируема через API или интерфейс
3. Не переходить к следующему этапу без завершения текущего
4. Минимизировать преждевременное усложнение (например, не внедрять сложные системы до MVP)

---

# 7. Ключевой ориентир

Первый значимый результат:

> Приложение открывается на мобильном устройстве, пользователь добавляет транзакцию, и она сохраняется через backend.

Это является точкой, после которой можно переходить к масштабированию функционала и подготовке к публикации.

---

Если требуется, следующим шагом можно подготовить:

* структуру репозитория
* пример docker-compose
* пример GitHub Actions workflow
* базовую структуру Flutter приложения

в виде конкретных файлов и конфигураций.
